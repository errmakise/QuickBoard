\chapter{绪论}\label{chap:introduction}

\section{研究背景与意义}
在线教学与远程协作逐渐常态化，可视化表达与即时讨论的需求持续增长。数字白板及相关协作工具在教育与专业协作领域的应用范围不断扩展，市场研究报告也反映出该类工具的持续增长趋势\cite{market_vmr_2024,market_htf_2025}。与传统文档协作相比，白板应用更适合承载草图、流程图与推导过程，但其工程实现面临更高频的输入事件与更复杂的可视化表达，渲染与网络开销在对象规模增长时更易放大。

在产品形态上，平台化白板往往引入账号体系、团队空间与复杂权限配置，适用于长周期与强流程的协作；而短会话场景（例如课堂推导、临时答疑、小组讨论与即时头脑风暴）更强调进入路径短、表达成本低与反馈及时。此类场景对“无需注册、快速创建/加入房间、弱网可恢复”的体验更敏感。基于上述需求，设计并实现一个聚焦核心协作链路、可交付且可验证的轻量协作白板系统具有现实意义。

\subsection{数学场景的表达痛点}
在教学与工程讨论中，数学公式、符号推导与示意图经常需要在同一画布中混合表达。纯 LaTeX 输入具有较高学习成本，且在即时讨论中会打断思路；而将手写内容以截图形式保留虽然门槛低，但不可编辑、难复用，难以沉淀为可检索的知识材料。对短会话协作而言，理想的公式输入链路应当满足两点：其一，输入侧接近板书（手写或自由绘制）以降低门槛；其二，输出侧可编辑可复用（可改写 LaTeX 并稳定渲染），从而实现内容沉淀。QuickBoard 将公式识别与可编辑渲染作为系统差异化能力，并以“可定位失败原因”的工程闭环提升可用性。

\subsection{短会话协作的工程挑战}
短会话协作往往呈现“快速进入—高频输入—随时离开”的模式，工程上带来三个挑战：第一，输入事件高频且连续，若缺少过程链路降载，网络与服务端转发容易成为瓶颈；第二，参与者可能随时断线或切换网络，若缺少版本补包与快照对齐机制，状态分叉难以恢复；第三，画布对象不断累积会放大渲染开销，影响平移缩放与选择操作。QuickBoard 的设计围绕这三类挑战展开：用 Ghost Brush 隔离过程预览、用服务端权威裁决与版本对齐保证最终收敛、用视口裁剪与对象缓存控制渲染成本。

\section{研究现状与相关工作}
\subsection{实时协作一致性技术}
多人协作的关键问题是并发编辑冲突与一致性收敛。OT（Operational Transformation）通过操作转换保持意图并实现一致性，是早期协同编辑系统的重要技术路线\cite{sun_ellis_1998}。CRDT（Conflict-free Replicated Data Type）从可交换、可合并的数据类型角度实现最终一致性，其理论与工程实践在近年来得到广泛研究\cite{shapiro_rr7506_2011}。OT 与 CRDT 各有适用场景：OT 在中心化协作中具备较好的意图保持能力，但通常依赖中心服务器与转换函数；CRDT 适配分布式与离线合并，但数据结构、存储与调试成本更高。针对图形对象集合模型，工程上也存在折中路径，即在明确业务边界内采用更轻量的合并策略，并结合服务端权威裁决与对齐兜底保证最终可用。

\subsection{协作白板产品与开源实现}
主流商业白板产品（例如 Miro、Mural、Microsoft Whiteboard 等）功能完整，适合长周期协作，但对短会话即时讨论并不总是友好。开源白板以 Excalidraw 为代表，交互体验较轻量并支持协作\cite{excalidraw}，适合作为教学演示与二次开发基础。不同系统在同步策略、弱网鲁棒性与扩展能力（如公式识别闭环）上存在差异。本课题以 QuickBoard 为载体，围绕“最终收敛、性能可用、公式闭环可定位”开展工程实现与验证。

\subsection{公式识别相关工作概述}
数学公式识别通常涉及图像预处理、结构建模与符号序列生成等问题。与印刷体相比，手写公式存在笔迹差异大、符号粘连、上下标密集与多行结构复杂等特点，使得识别更易出现空结果或结构错误。工程系统在集成识别能力时，需要重点处理两类落地问题：其一，输入端如何提供稳定且信息密度合理的截图（避免裁剪过紧或噪声过多）；其二，输出端如何将识别结果纳入“可编辑、可纠错”的交互闭环，而不是一次性输出并直接落盘。QuickBoard 的公式能力以工程闭环为核心：识别结果进入确认与编辑流程，失败通过统一错误码与 traceId 进行定位，并允许用户在插入前修正内容，以提升实际可用性。

\subsection{轻量系统的工程取舍}
从工程交付角度看，毕业设计需要在创新性、可实现性与可验证性之间平衡。本课题将创新点聚焦为“弱网最终收敛 + 性能与降载 + 公式闭环”，并将验证材料作为论文写作的重要组成部分。对应地，系统不引入账号体系与复杂权限，不追求覆盖全部企业级白板功能，而是优先保证核心链路稳定，并将实验条件、指标与证据路径固化为可复核报告。这一取舍有助于将论文从“功能罗列”转化为围绕关键问题的论证结构。

\section{研究内容与主要工作}
围绕 QuickBoard 系统，本文完成的主要工作包括：
\begin{itemize}
  \item 设计并实现房间化的协作入口，支持创建房间、加入房间与最近房间管理，降低短会话进入成本；
  \item 实现实时协作通信与同步策略：采用服务端权威裁决与字段级 LWW 合并，并结合 roomVersion 增量补包、快照兜底与手动对齐保证弱网与断线场景下的最终收敛；
  \item 实现渲染与网络侧优化：通过视口裁剪与对象缓存提升大对象规模下的渲染性能；通过 Ghost Brush 将高频过程预览与权威状态解耦并采用批处理/限包降低消息量；
  \item 集成数学公式识别：通过后端代理调用 OCR 服务并统一错误码与 traceId，前端实现“识别—确认插入—可编辑渲染”的闭环交互。
\end{itemize}

\section{研究目标与问题定义}
结合短会话协作与教学场景的实际需求，本课题将研究目标收敛为四个可落地问题：
\begin{enumerate}
  \item \textbf{如何在弱网与断线重连下保证最终可用的一致性？} 该问题强调“最终收敛与可对齐”，并要求裁决规则可解释、实现复杂度可控。
  \item \textbf{如何控制对象规模增长下的渲染开销？} 该问题强调在对象数量累积后仍可完成平移、缩放、选择与编辑等基本交互，并用基准数据量化优化效果。
  \item \textbf{如何降低高频输入造成的网络压力？} 该问题强调过程预览与权威状态分离，通过批处理与限包避免消息风暴挤占权威增量带宽。
  \item \textbf{如何构建低门槛的公式输入闭环？} 该问题强调从手写到可编辑公式的链路完整，并要求失败可定位、可纠错，避免出现“识别失败无从下手”的体验断点。
\end{enumerate}
围绕上述问题，QuickBoard 的设计与实现以工程交付为导向，通过对照实验、基准测试与统计验证形成可复核证据链，使论文论证从功能描述转化为“问题—方法—验证”的结构化表达。
