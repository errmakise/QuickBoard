\chapter{测试与评估}\label{chap:eval}

本章从功能正确性、工程可交付性与关键链路稳定性三个角度对 QuickBoard 进行测试与评估，并给出结果分析。

\section{测试策略}
项目采用“单元测试 + 构建验证 + 场景化冒烟测试”的组合策略：
\begin{itemize}
  \item \textbf{单元测试}：覆盖协作通信、OCR 请求与裁剪、同步逻辑等核心模块，降低迭代带来的回归风险。
  \item \textbf{构建验证}：确保生产构建通过，避免仅开发环境可运行的问题。
  \item \textbf{场景化测试}：以典型协作流程（创建/加入、多人同步、断线重连、公式识别与编辑、导出）进行端到端验证。
\end{itemize}

\section{单元测试与构建结果}
前端使用 Vitest 执行单元测试，并在每轮关键改动后执行构建验证。测试与构建输出以报告形式归档在 \texttt{doc/test\_reports}，便于回溯与论文撰写引用。

\section{典型场景测试}
选取两个浏览器窗口模拟两名用户，进行如下流程验证：
\begin{enumerate}
  \item 创建房间并复制链接，另一端通过链接加入；
  \item 进行画笔绘制与形状绘制，验证对端实时同步与撤销行为；
  \item 断开网络后重连，验证状态恢复与同步连续性；
  \item 框选区域进行公式识别，确认后插入画布并双击编辑；
  \item 导出画布内容并检查导出文件可用性。
\end{enumerate}
上述流程覆盖了系统的主链路与关键交互点，可用于答辩演示与论文截图。

\section{结果分析与不足}
从测试结果看，QuickBoard 能够满足“快速进入协作、实时同步绘制、公式识别与可编辑显示”的核心目标。其不足主要体现在：（1）协作一致性策略以工程可用为主，未覆盖强一致性的复杂合并需求；（2）OCR 识别质量受输入质量影响较大，在复杂背景或笔迹较淡时仍可能失败；（3）在极端多对象场景下渲染性能仍受限于 Canvas 与对象模型的固有开销。
